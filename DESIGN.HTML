Slate interface language

Show the work.
Quiet everything else.

A plain-English visual guide for building a compact, trustworthy expert-agent platform. It turns the repository’s existing tokens and operational patterns into practical decisions.

The user’s task belongs in the first viewport

Start here

Five rules carry most of the design system. Use them before reaching for a new component or visual treatment.

1

Put work first

A title and short orientation should lead directly to the working surface.

2

Stay Slate

Dark flat surfaces, blue action and focus, semantic colors for real state.

3

Match size to value

Small context gets small space. Important decisions get direct controls.

4

Group decisions

Use one organized surface with dividers instead of a collection of cards.

5

Explain on demand

One useful sentence now; rare detail when the user reaches uncertainty.

Good / avoid

The visual language should feel like an operational console, never an in-app marketing page.

Good

Compact, flat, direct

  • Actual task visible immediately
  • Blue marks action, focus, and selection
  • Short labels in sentence case
  • One surface per related decision set
  • Warnings only when the user can act
Avoid

Decorative hierarchy

  • Hero copy inside application routes
  • Purple page-specific branding
  • Large cards for summary numbers
  • Gradients, glow, and oversized radii
  • Idle floating controls or duplicate overview pages

Codex prompts

Copy these when you want future UI work to stay aligned with this system.

General UI change

Build this as a compact operational interface using the repository's existing Slate design tokens from client/src/App.css. Put the user's actual task in the first viewport. Apply the space/value assessment from DESIGN.md to every visible element. Prefer one organized surface with dividers over a collection of cards. Verify desktop and mobile in the live app.

Settings change

Keep the Settings page header under roughly 72px and start the working surface within roughly 150px of the content top unless a real critical warning requires more room. Do not add decorative eyebrow copy, hero text, large summary cards, gradients, or purple accents. Check loading, empty, warning, error, disabled, dirty, and saved states.

Appendix

Implementation notes and durable quality checks.

Spacing and component defaults
Use a 4px rhythm. Common values are 4, 6, 8, 10, 12, 14, 16, 20, 24, and 32px. Panels use an 8px radius and 12–16px padding. Rows use 7–10px padding. Controls are usually 30–36px tall. In-flow panels do not need shadows.
Accessibility
Preserve visible keyboard focus, real semantic controls, accessible names for icons, non-color status labels, reduced-motion support, readable contrast, and scrollable mobile navigation with a visible selected state.
Known design debt
The client has several large CSS layers and legacy high-specificity overrides. Settings currently reconciles older Account and AI Safety styles into Slate. Future consolidation should reduce override depth without treating older Apple-grey or purple styling as precedent.
Release check
Name the primary user task; assess every first-viewport element; reuse tokens; inspect all states; build; verify desktop and mobile live; check console errors; and re-read changed files before reporting.
