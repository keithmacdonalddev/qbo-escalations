1. Canonical Intake Reliability

This means: before the app reasons about an escalation, it needs to preserve the original evidence and the first clean structured version of the case.

Plain English version:

When a screenshot or conversation becomes an escalation, the app should always remember:

- the original screenshot or source conversation
- the raw parsed text
- the structured parser fields
- parser validation issues
- which provider/model parsed it
- which provider/model triaged it
- what confidence or warnings existed
- whether the parser succeeded, partially succeeded, or failed

Why this matters:

If the app later says, “This looks like a Payroll T4 XML issue,” we need to know why. Did the screenshot actually say that? Did the parser infer it? Did an AI model
guess it? Did a human correct it?

Canonical intake is the root evidence record. If that layer is weak, every later layer becomes questionable: triage, INV matching, escalation records, resolution, and
knowledge drafts.

So this is less about adding a new feature and more about making sure the first version of the case is trustworthy.

2. Case Lifecycle Consistency

The lifecycle I described was:

parsed -> triaged -> analyst answered -> escalation record linked -> resolved/escalated -> knowledge candidate

That means a case should move through understandable states instead of becoming scattered across chat messages, parser output, an escalation record, and maybe a
knowledge draft.

And yes, your instinct is right: this should probably become a notification center or review queue.

Something like:

Needs Attention

- Parser failed validation
- Triage used fallback
- Strong INV candidate needs confirmation
- Escalation record has not been linked
- Case is resolved but missing resolution notes
- Knowledge draft is ready for review
- Knowledge draft was approved but not published
- Possible duplicate case detected

This would be the practical UI bridge between the current app and the future ontology system.

Instead of the ontology being invisible backend theory, the app would start saying:

> “Here are the cases that need a decision from you.”

That is much more useful.

3. Duplicate And Retry Safety

This means the app should avoid creating duplicate records when the same thing is retried.

Example:

You parse a screenshot. The parser fails. You retry. Then you send it into chat twice. The app should not accidentally create:

- two escalation records
- two knowledge drafts
- two “known issue” links
- two review tasks

The app has some partial protection now, but not a complete system.

From the current implementation, there are signs of safety around some paths:

- a conversation generally avoids auto-creating another escalation if it already has an escalationId
- screenshot uploads appear to track hashes so duplicate screenshot files can be skipped inside an escalation
- KnowledgeCandidate is unique per escalation, so one escalation should not normally create multiple knowledge drafts

But that does not fully solve the bigger problem.

The weak spot is this:

same real-world case -> different conversation or retry path -> duplicate Escalation records

If duplicate escalations are created, then duplicate knowledge candidates can follow because each escalation looks separate.

So I would add a lightweight duplicate check around things like:

- COID
- case number
- screenshot hash
- parsed actual outcome
- category
- conversation source
- time window

Not to block everything automatically, but to warn:

> “This looks similar to an existing escalation. Link to existing case instead?”

Who Determines What Goes Into A Knowledge Draft?

Right now, conceptually, there are three contributors:

1. The escalation fields
   Things like category, attempting to, actual outcome, troubleshooting steps, case number, COID.
2. The resolution state
   What fixed it, why it was resolved, whether it was escalated further, what notes were added.
3. Optional AI enrichment
   The app can use a model to turn the escalation into cleaner knowledge draft fields like title, symptom, root cause, exact fix, signals, and summary.

But the important part is this:

The human should determine whether it becomes reusable knowledge.

AI can draft. The app can suggest. But publishing to a playbook or ontology should require review.

That is why knowledge candidates need strong labels like:

- safe reusable pattern
- edge case
- customer-specific
- temporary incident
- unsafe to reuse
- case history only

The app already has some of this shape. The hardening work would make it stricter and more evidence-backed.

4. Known Issue / INV Confidence Boundaries

This one is important.

A matching INV should not automatically mean:

> “This is the known issue.”

It should mean:

> “This might be related to a known issue.”

That distinction matters because known issue matching can be dangerous if overstated.

Example:

A case says:

Customer cannot file T4 XML.

There may be an existing INV about T4 XML export failures.

That is a candidate match.

But it is only a strong match if the evidence lines up:

- same product area
- same workflow
- same error or symptom
- same timing
- same affected form/report
- same workaround or known behavior
- same region/year if relevant

A weak match might be:

- both mention payroll
- both mention tax
- both mention filing
- both are in the same broad category

That is not enough to confidently say it is the same issue.

So “confidence boundaries” means the app should preserve the difference between:

Strong candidate:
The symptom, workflow, product area, and error text match.

Weak candidate:
Only the category and a few terms match.

Rejected candidate:
The app suggested it, but the human said it is not related.

Confirmed match:
The human or strong evidence confirmed it.

Why persist that?

Because future AI should not come back later and say:

> “This was caused by INV-12345”

when the actual truth was:

> “INV-12345 was only a weak candidate and was never confirmed.”

This is ontology-relevant because the ontology should learn relationships carefully. Bad relationship data is worse than no relationship data.

5. Resolution Discipline

You already agreed with this one, and I think it is one of the highest-value hardening rules.

A case should not be “resolved” with no resolution.

Bad:

status: resolved
resolution: empty
resolution notes: empty

Good:

status: resolved
resolution: Customer corrected payroll item mapping and regenerated T4 XML successfully.
resolution notes: Issue was not an active INV. Parser initially suggested INV match but it was rejected.

Why this matters:

Knowledge extraction depends on the resolution. Without it, the app can only learn the symptom, not the fix.

So I would enforce something like:

- resolved requires a resolution summary or reason
- escalated-further requires an escalation reason/path
- rejected knowledge candidate requires review notes
- unsafe-to-reuse requires a reason

This is not bureaucracy. It is how the app avoids producing weak reusable knowledge.

6. Knowledge Candidate Safety

This means the app should not treat every resolved case as reusable knowledge.

A resolved case can mean different things:

Reusable pattern:
This is a general fix that applies to future customers.

Edge case:
This is useful, but only in a narrow situation.

Customer-specific:
This was caused by one customer’s setup.

Temporary incident:
This was caused by an outage or active known issue.

Unsafe to reuse:
The answer depended on incomplete evidence or uncertain assumptions.

Case history only:
Keep it searchable, but do not teach the playbook from it.

Where I would implement this:

Server side:

- strengthen KnowledgeCandidate
- preserve source evidence
- preserve parser/triage/INV/resolution lineage
- prevent unsafe publishing
- prevent duplicate candidates
- require resolution evidence before draft generation

UI side:

- improve the knowledge review panel
- add clearer “allowed use” decisions
- show what evidence the draft came from
- show warnings when root cause or exact fix is missing
- make approval/publish a deliberate action

Workflow side:

- add review queue items when a knowledge draft needs human attention
- add warnings when a draft was generated from fallback triage, weak INV evidence, or incomplete parser output

The full implementation shape would be:

1. Intake happens
   Screenshot/conversation is parsed.
   Raw text, fields, provider/model, validation issues are preserved.

2. Triage happens
   Triage card is created.
   App records whether it was confident, partial, or fallback.

3. Known issue search happens
   INV candidates are recorded as candidates.
   App records why they matched and whether they were accepted/rejected.

4. Escalation record is linked
   The case gets a durable escalation record.
   Duplicate checks prevent accidental copies.

5. Analyst works the case
   Chat response, notes, and decisions attach to the case.

6. Case is resolved or escalated further
   Resolution summary or escalation reason is required.

7. Knowledge candidate is generated
   Draft includes source snapshot and evidence trail.
   It is not automatically reusable knowledge.

8. Human reviews it
   Human decides:
   canonical, edge case, case history only, customer-specific, temporary incident, unsafe to reuse.

9. Approved knowledge can publish
   Only safe approved drafts become playbook/ontology material.

That is the safe version.

Evidence Ledger

An evidence ledger is just a structured timeline of important workflow events.

It is not a blockchain. It does not need to be complicated.

Think of it as receipts.

Example:

parse.completed

- screenshot hash
- raw parsed text
- structured fields
- parser model
- validation issues

triage.completed

- severity
- category
- confidence
- model used
- fallback used: false

inv.match.proposed

- INV-12345
- match strength: weak
- reason: same category, similar symptom
- status: needs human confirmation

inv.match.rejected

- rejected by user
- reason: different product behavior

escalation.linked

- conversation id
- escalation id

resolution.recorded

- status: resolved
- resolution summary
- resolved at

knowledge.draft.generated

- candidate id
- source escalation id
- generated from resolution + triage + parser evidence

knowledge.approved

- approved by user
- reusable outcome: edge case

Why this matters:

The future ontology should not just know:

T4 XML issue -> fixed by X

It should know:

This relationship came from a resolved escalation,
with parser confidence high,
INV match rejected,
human-reviewed knowledge approved as edge case.

That is much more trustworthy.

Ontology MVP Trail

Yes. That is exactly how I see it.

This would be an MVP trail to feel what early ontology implementation might be like without building the whole ontology platform.

The first version should feel like:

- the app knows the state of each escalation
- the app tells you what needs attention
- the app preserves why decisions were made
- the app prevents weak cases from becoming strong knowledge
- the app turns resolved cases into reviewable knowledge candidates
- the app starts building structured relationships from real work

The ontology slice should not initially feel like a new separate product. It should feel like the current QBO escalation app got smarter, more organized, and more
careful.

A good first planning target would be:

QBO Escalation Review Queue + Evidence Ledger + Safer Knowledge Candidate Flow

That gives you an early practical version of the ontology idea while keeping it tied to real escalation work.
