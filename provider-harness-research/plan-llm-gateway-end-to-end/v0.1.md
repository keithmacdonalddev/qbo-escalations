# LLM Gateway Provider Harness End-to-End Plan v0.1

Date: 2026-05-21

## Executive Decision

This plan continues the provider-by-provider direction.

```text
Provider: LLM Gateway
App provider id: llm-gateway
Research id: llm-gateway
Goal: make this one provider harness deterministic end to end before bringing another provider up to the same standard.
```

This is not a parser-quality plan. This is not a prompt plan. This is not a fallback plan. This is not a UI plan.

The only goal is:

```text
LLM Gateway provider attempt starts
-> this app sends the LLM Gateway request package
-> LLM Gateway sends a response/error package
-> this app receives that exact package at the provider boundary
-> Mongo preserves that qbo-visible package in a gateway-specific shape
-> existing app behavior continues unchanged
```

## Plain-English Definition

A provider harness is the plumbing around one provider call.

For this plan, the provider is `llm-gateway`, meaning the local service running separately at:

```text
C:\Projects\llm-gateway
```

The qbo-escalations app talks to that service over HTTP, usually at:

```text
http://127.0.0.1:4100
```

The gateway may then talk to LM Studio behind the scenes at:

```text
http://127.0.0.1:1234
```

Those are two different boundaries.

This plan is for the qbo-escalations boundary only:

```text
qbo-escalations -> llm-gateway -> qbo-escalations
```

This plan does not capture the hidden upstream boundary inside the gateway:

```text
llm-gateway -> LM Studio -> llm-gateway
```

If we later want the gateway process to preserve the raw upstream LM Studio package too, that is a separate harness inside `C:\Projects\llm-gateway`. Do not mix that work into this qbo-side plan.

## Non-Negotiable Rules

1. `llm-gateway` is the only provider in scope.
2. The qbo-visible gateway package is the only package in scope.
3. The hidden upstream LM Studio package inside `C:\Projects\llm-gateway` is out of scope for this qbo-side plan.
4. Every in-scope LLM Gateway provider attempt must create one gateway package record when capture is enabled and Mongo is connected.
5. The record must preserve the provider package, not just the extracted answer text.
6. Provider calls must not wait for Mongo/file recording before returning the provider result to existing app code.
7. Recorder writes are fire-and-forget in production paths.
8. Tests may use the existing test-only recorder-settled hook.
9. Recorder failure must not fail the provider call.
10. Recorder failure must leave enough server-side evidence to diagnose capture failure.
11. No parser validation changes.
12. No model answer cleanup changes.
13. No prompt changes.
14. No fallback changes.
15. No UI changes.
16. No provider selection changes.
17. No changes inside `C:\Projects\llm-gateway` for this qbo-side plan.
18. No direct LM Studio harness changes unless a shared helper needs a tiny compatibility update and existing LM Studio tests still pass.
19. No provider-harness terminal output may include full prompt text, base64 images, raw provider bodies, API keys, bearer tokens, or cookies.
20. Do not add temporary color console logs as a completion requirement. Verification should come from captured Mongo records, focused tests, and explicit read-only inspection/export commands.

## What "100%" Means Here

For this plan, "Mongo gets everything LLM Gateway sent to qbo-escalations" means:

- the full HTTP status and status message when an HTTP response exists
- response headers, including `x-request-id`
- raw response headers in original order
- response trailers and raw trailers when present
- ordered raw body chunks
- full response body text
- parsed JSON view when parseable
- JSON parse error when not parseable
- full non-200 error body, not only the current truncated `Error.message`
- gateway error envelope when present
- gateway-added metadata object when present
- gateway request id copied from response headers for lookup convenience
- network error facts when no HTTP response arrives
- timeout facts
- abort/cancel facts
- request method, URL, path, headers, timeout, body text, body JSON, body byte length, and body digest
- image data URL request payloads inline or by explicit external reference

The raw provider response body must not be transformed into canonical text at this layer.

Allowed safety exceptions:

- outgoing `Authorization` headers must be redacted before durable storage
- outgoing `Proxy-Authorization` and cookie-like headers must be redacted if ever present
- secret-like request body keys must be redacted if ever present

Payload-size rule:

- if a request or response payload is too large for safe Mongo inline storage, store a digest and external payload reference
- the record must explicitly show where the payload was stored
- no silent truncation
- no losing provider-sent response bodies

### Top-Level Record

Expected top-level fields:

```js
{
  schemaVersion: '0.1',
  captureVersion: 'provider-harness-llm-gateway-v0.1',
  providerId: 'llm-gateway',
  providerResearchId: 'llm-gateway',
  providerPathType: 'gateway-http',
  callSite: 'image-parser:callLlmGateway' | 'remote-api-providers:requestLlmGatewayChat' | 'image-parser:validateRemoteProvider:llm-gateway',
  operation: 'image-parse' | 'chat' | 'provider-status',
  source: { file, functionName, helperName },
  request: <optional generic compatibility view>,
  response: <optional generic compatibility view>,
  llmGateway: <strict gateway package>,
  timing: <shared timing>,
  outcome: 'success' | 'http_error' | 'network_error' | 'timeout' | 'aborted' | 'invalid_json',
  error: <shared compact error>,
  redaction: <shared redaction notes>,
  storage: <shared storage refs>
}
```

### `llmGateway.request`

Required request fields:

```js
{
  method: 'POST' | 'GET',
  baseUrl: 'http://127.0.0.1:4100',
  url: 'http://127.0.0.1:4100/v1/chat/completions',
  protocol: 'http:',
  hostname: '127.0.0.1',
  port: 4100,
  path: '/v1/chat/completions',
  urlPath: '/v1/chat/completions',
  headers: {},
  redactedHeaderNames: [],
  bodyKind: 'none' | 'text' | 'json',
  bodyText: string | null,
  bodyTextPayloadRef: payloadRef | null,
  bodyJson: object | null,
  bodyJsonPayloadRef: payloadRef | null,
  bodyByteLength: number,
  bodySha256: string | null,
  modelRequested: string,
  stream: false,
  timeoutMs: number | null,
  hasImages: boolean,
  images: []
}
```

Image entries for image-parser calls:

```js
{
  seq: number,
  mediaType: 'image/png' | 'image/jpeg' | string,
  source: 'data-url',
  dataUrlSha256: string,
  decodedByteLength: number | null,
  dataUrlPayloadRef: payloadRef | null
}
```

The image data URL itself is part of the exact request body. It may be externalized, but the record must point to it and include a digest.

### `llmGateway.response`

Required response fields:

```js
{
  received: boolean,
  statusCode: number,
  statusMessage: string,
  httpVersion: string,
  headers: {},
  redactedHeaderNames: [],
  rawHeaders: [],
  trailers: {},
  rawTrailers: [],
  gatewayRequestId: string | null,
  bodyChunks: [],
  bodyText: string,
  bodyTextPayloadRef: payloadRef | null,
  bodyByteLength: number,
  bodySha256: string | null,
  parsedJson: object | null,
  parsedJsonPayloadRef: payloadRef | null,
  jsonParseError: {
    name: string,
    message: string
  } | null
}
```

Body chunk entries:

```js
{
  seq: number,
  receivedAt: string,
  byteLength: number,
  sha256: string | null,
  text: string | null,
  textPayloadRef: payloadRef | null
}
```

### `llmGateway.gateway`

This section preserves the provider-specific data the gateway adds or exposes.

```js
{
  requestId: string | null,
  metadata: object | null,
  usage: object | null,
  cost: object | null,
  credits: object | null
}
```

Mapping rules:

- `requestId` comes from `response.headers['x-request-id']` when present.
- `metadata` is `parsedJson.gateway` exactly as returned.
- `usage` is `parsedJson.gateway.usage` when present.
- `cost` is `parsedJson.gateway.cost` when present.
- `credits` is `parsedJson.gateway.credits` when present.

Do not require `credits`; gateway docs/source indicate credits can be absent depending on key type.

### `llmGateway.error`

Required for failure outcomes:

```js
{
  name: string,
  message: string,
  code: string,
  stack: string,
  statusCode: number | null,
  rawBody: string | null,
  rawBodyPayloadRef: payloadRef | null,
  object: object | null,
  gatewayErrorType: string,
  gatewayErrorCode: string,
  gatewayErrorMessage: string,
  nodeErrorCode: string
}
```

For HTTP errors, preserve the full gateway error envelope from the response body when parseable.

For network errors, preserve Node error facts even when there is no HTTP response body.

### `llmGateway.providerStatus`

For `operation: 'provider-status'`, optionally denormalize a small qbo-visible view while still preserving the full `parsedJson`:

```js
{
  ok: boolean | null,
  provider: string,
  authenticated: boolean | null,
  upstream: object | null,
  gateway: object | null,
  error: object | null
}
```

This is only a convenience view. The full provider-status body remains in `llmGateway.response.bodyText` and `llmGateway.response.parsedJson`.
