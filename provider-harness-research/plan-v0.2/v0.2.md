# Provider Harness v0.2 Plan

## Executive Decision

v0.2 is CLI-only.

The goal is to capture the provider package from local CLI model calls without changing provider behavior, parser behavior, fallback behavior, or UI behavior.

Implementation order:

1. Codex CLI first.
2. Stop and review.
3. Claude CLI second, only after Codex is stable.

This version does not include SDK capture, streaming/SSE capture, full temp-file preservation, dashboards, provider health checks, warm-ups, key probes, model discovery, parser validation, answer cleanup, or canonical text conversion.

## Plain-English Goal

The app calls model providers in different ways. v0.1 captured plain HTTP model calls. v0.2 should capture local command-line provider calls.

For CLI providers, the package that reaches this server is:

```text
command name
args
stdin/prompt package
stdout
stderr
exit code
signal
timeout/error facts
timing
```

The harness preserves that package so the app can later answer:

```text
What exactly came back from this provider path?
```

It must not try to answer:

```text
Was the model answer correct?
```

## v0.1 Context

v0.1 created the HTTP capture foundation:

- `server/src/models/ProviderCallPackage.js`
- `server/src/services/provider-call-package-recorder.js`

v0.1 captured direct/non-streaming HTTP provider packages for:

- `anthropic`
- `openai`
- `gemini`
- `kimi`
- `llm-gateway`
- non-streaming `lm-studio`

v0.1 did not capture:

- CLI subprocess stdout/stderr packages
- Anthropic Agent SDK message objects
- streaming/SSE frames
- provider warm-ups
- provider health checks
- key-test probes
- model discovery
- parser validation
- canonical text conversion

v0.2 extends the same preservation idea to CLI subprocesses only.

## Non-Negotiable Rules

1. Capture must be invisible to provider behavior.
2. Capture must be disabled unless `ENABLE_PROVIDER_CALL_PACKAGE_CAPTURE=true`.
3. Provider calls must not wait for Mongo/file recording before returning the provider result to existing code.
4. Recorder writes are fire-and-forget in production paths.
5. Recorder failures must not fail the provider call.
6. Recorder failures must be returned as structured recorder results and covered by tests; they should not add development console breadcrumbs.
7. Tests may use a test-only recorder-settled hook.
8. No parser logic changes.
9. No answer quality decisions.
10. No prompt changes.
11. No fallback behavior changes.
12. No UI changes.
13. No temp-file preservation hardening in v0.2.
14. Do not add provider-harness development console breadcrumbs. Verification should come from captured records, focused tests, and explicit read-only inspection commands.

## Recorder Policy

This is a hard v0.2 rule:

```text
provider call completes
-> existing provider code returns/resolves/calls callbacks as it does today
-> recorder persists the package in the background
```

Production code must not do this:

```js
await recordProviderCallPackage(...)
```

Production code should do this:

```js
void recordProviderCallPackage(...);
```

The recorder must catch its own errors and return a structured result. A Mongo failure, payload-store failure, redaction failure, or serialization failure must not change the provider call result.

## Console Trace Policy

Provider-harness development console breadcrumbs were removed after v0.1/v0.2 testing. Do not reintroduce `providerHarnessTrace(...)`, `PROVIDER_HARNESS_CONSOLE_TRACE`, `picocolors`, or raw `console.log(...)` breadcrumbs for normal provider-harness flow.

Verification should use deterministic evidence instead:

- captured `ProviderCallPackage` records.
- sidecar payload files and hashes.
- focused tests.
- explicit read-only Mongo/file inspection commands during manual testing.

Operational warnings may be added only if they are part of normal application error handling, not temporary provider-harness path visualization.

### Test Hook

Tests may use a test-only hook to wait for background recording:

```js
await __waitForProviderPackageRecorderSettled();
```

Rules:

- The hook is test-only.
- It must not be required by production code.
- It exists only so tests can assert the background record after provider behavior has completed.

## Data Model Change

Current `ProviderCallPackage` has HTTP-oriented top-level fields:

- `request`
- `response`
- `timing`
- `outcome`
- `error`
- `redaction`
- `storage`

v0.2 must add one CLI package field:

```js
cli: { type: mongoose.Schema.Types.Mixed, default: null }
```

Do not add `sdk` or `stream` fields in v0.2 unless that future work is moved back into scope.

Do not add indexes for `cli` fields in v0.2.

Use:

```text
captureVersion: provider-harness-cli-v0.2
providerPathType: cli
```

Use the actual app provider ids:

- `codex`
- `claude`

Use research labels only in `providerResearchId`:

- `openai-cli`
- `anthropic-cli`

## CLI Package Shape

Recommended `cli` subdocument:

```js
cli: {
  command: 'codex',
  args: [],
  cwd: {
    value: null,
    redacted: true
  },
  env: {
    capturedKeys: [],
    notes: []
  },
  stdin: {
    text: null,
    byteLength: 0,
    sha256: null,
    externalRef: null
  },
  stdout: {
    text: '',
    byteLength: 0,
    sha256: null,
    lines: [],
    jsonlEvents: [],
    malformedLines: [],
    finalBuffer: '',
    externalRef: null
  },
  stderr: {
    text: '',
    byteLength: 0,
    sha256: null,
    externalRef: null
  },
  process: {
    pid: null,
    exitCode: null,
    signal: null,
    spawned: false,
    closed: false,
    killed: false,
    killSignal: null
  },
  timeout: {
    timeoutMs: null,
    fired: false
  }
}
```

### Stdout Rules

`stdout.text` is the raw stdout text.

`stdout.lines` is the line-split view of `stdout.text`.

`stdout.jsonlEvents` is the parsed-object view of parseable JSONL lines.

`stdout.malformedLines` contains raw lines that failed JSON parse.

`stdout.finalBuffer` contains any trailing non-newline-terminated text.

Do not treat JSONL parse failure as parser failure. It is provider package evidence.

### Stderr Rules

`stderr.text` is the raw stderr text.

Do not classify stderr as failure by itself. Some CLIs may write informational output to stderr.

Failure is decided by the outcome rules below.

### Large Payload Rules

Avoid storing the same large payload multiple times inline.

Allowed inline values:

- small stdin text.
- small stdout text.
- small stderr text.
- small parsed event arrays.

When values are large, store:

- byte length.
- sha256.
- external reference.
- a short inline preview if useful.

Do not design GridFS, retention, dashboard indexes, or storage engines in v0.2.

## Outcome Rules

Each CLI provider attempt gets exactly one outcome.

Decision tree:

1. If the child process could not be spawned, use `spawn_error`.
2. If an abort signal fired, use `aborted`.
3. If the timeout fired, use `timeout`.
4. If the process emitted an error event, use `process_error`.
5. If the process closed with nonzero exit code, use `process_error`.
6. If the path expected JSONL and stdout had no parseable JSONL events, use `invalid_jsonl`.
7. Otherwise use `success`.

Allowed v0.2 CLI outcomes:

- `success`
- `spawn_error`
- `process_error`
- `timeout`
- `aborted`
- `invalid_jsonl`

Recorder failures are not provider attempt outcomes. They should be logged by the recorder without changing the provider attempt outcome.

## Redaction Rules

Redact before persistence.

Always redact:

- API keys.
- bearer tokens.
- cookies.
- local auth/session paths.
- home-directory specific paths when not needed.
- secret-like env values.

For CLI env capture:

- record only env keys that materially affect the subprocess.
- do not store arbitrary environment values.
- do not store `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, or similar secret values.

For CLI args:

- preserve command shape.
- redact sensitive path segments or token-like args.
- preserve enough information to understand what the subprocess was asked to do.

For stdin:

- preserve the prompt/package text because that is part of the provider request package.
- redact secrets if found.
- externalize when large.

## Payload Externalization Paths

The payload store must know which CLI fields can become large.

v0.2 large-value candidates:

- `cli.stdin.text`
- `cli.stdout.text`
- `cli.stdout.lines`
- `cli.stdout.jsonlEvents`
- `cli.stdout.malformedLines`
- `cli.stderr.text`

Do not externalize tiny values just to make records harder to inspect.

Do externalize before the Mongo document risks exceeding size limits.

## Test Requirements

Required test groups:

1. Capture disabled:
   - no record is written.
   - provider behavior is unchanged.

2. Recorder fire-and-forget:
   - provider call completes without waiting for recorder persistence.
   - test-only hook can wait for the background write.

3. Successful Codex JSONL:
   - one package record is written.
   - stdout raw text is preserved.
   - line order is preserved.
   - parsed JSONL events are preserved.
   - stderr is preserved.
   - outcome is `success`.

4. Non-JSON stdout line:
   - raw line is preserved.
   - malformed line is preserved.
   - existing provider behavior is unchanged.

5. Nonzero exit:
   - exit code is preserved.
   - stderr is preserved.
   - outcome is `process_error`.

6. Timeout:
   - timeout facts are preserved.
   - signal/kill facts are preserved when available.
   - outcome is `timeout`.

7. Spawn error:
   - spawn error is preserved.
   - outcome is `spawn_error`.

8. Recorder failure:
   - provider result still returns normally.
   - recorder failure returns a structured recorder result or is observable through focused tests.
