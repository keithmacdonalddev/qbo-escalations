# Provider Harness v0.1 Final Implementation Plan

This is the consolidated v0.1 plan after reviewing:

- `provider-harness-research/plan-1/v.01.md`
- `provider-harness-research/plan-2/v.01.md`

The final direction is closest to plan 2, with several corrections from plan 1 and a fresh source check against the current repo.

## Executive Decision

Provider harness v0.1 should preserve the raw HTTP provider call package that reaches this server, without changing model selection, parser behavior, retries, fallback, prompts, answer cleanup, UI, dashboards, or inspection routes.

The shape is:

```text
server sends HTTP model request
-> provider response first reaches this server
-> helper captures the server-visible request/response package
-> secrets are redacted before persistence
-> oversized payloads are externalized with hashes and references
-> ProviderCallPackage record is written
-> existing provider code receives the same return value it receives today
```

This is a preservation layer, not an answer-quality layer.

## Success Criteria

v0.1 is successful when all of these are true:

1. A shared `ProviderCallPackage` Mongo model exists.
2. A shared recorder service exists.
3. A shared redaction helper exists.
4. A payload storage helper or explicit inline-size guard exists.
5. Capture is guarded by `ENABLE_PROVIDER_CALL_PACKAGE_CAPTURE`, defaulting to off.
6. One direct HTTP proof path is captured first.
7. Existing provider code receives the same return value it received before capture was added.
8. All in-scope HTTP model paths are then wired through the same capture pattern.
9. Mongo stores one `ProviderCallPackage` record per captured provider attempt when recording succeeds.
10. Request auth secrets are not stored.
11. Response headers are redacted defensively before persistence.
12. Raw response text and parsed JSON, when parseable, are preserved.
13. HTTP non-2xx responses are captured before existing provider code throws.
14. Network errors, timeouts, and aborts attempt non-fatal capture when possible.
15. Recorder failure does not fail, retry, or redirect the provider call.
16. No parser behavior changes.
17. No UI behavior changes.
18. No CLI, SDK, streaming, probe, warm-up, health-check, or model-discovery capture is added.

## Shared Contract: ProviderCallPackage

Use one Mongo model:

```text
ProviderCallPackage
```

Reason: the record preserves the whole provider call package, not only a provider response.

Recommended file:

```text
server/src/models/ProviderCallPackage.js
```

Recommended model style:

- explicit top-level metadata fields
- mixed subdocuments for request, response, error, redaction, storage, and timing
- timestamps enabled
- `versionKey: false`
- minimal indexes only
- no parser-answer validation fields
- no UI fields
- no analytics normalization

Recommended high-level schema:

```js
const providerCallPackageSchema = new mongoose.Schema({
  schemaVersion: { type: String, required: true, default: '0.1', index: true },
  captureVersion: { type: String, required: true, default: 'provider-harness-http-v0.1' },

  providerId: { type: String, required: true, index: true },
  providerResearchId: { type: String, default: '', index: true },
  providerPathType: { type: String, required: true, index: true },

  callSite: { type: String, required: true, index: true },
  operation: { type: String, required: true, index: true },
  source: { type: mongoose.Schema.Types.Mixed, default: null },

  request: { type: mongoose.Schema.Types.Mixed, default: null },
  response: { type: mongoose.Schema.Types.Mixed, default: null },
  timing: { type: mongoose.Schema.Types.Mixed, default: null },
  outcome: { type: String, required: true, index: true },
  error: { type: mongoose.Schema.Types.Mixed, default: null },
  redaction: { type: mongoose.Schema.Types.Mixed, default: null },
  storage: { type: mongoose.Schema.Types.Mixed, default: null },
}, {
  timestamps: true,
  versionKey: false,
  minimize: false,
});

providerCallPackageSchema.index({ createdAt: -1 });
providerCallPackageSchema.index({ providerId: 1, createdAt: -1 });
providerCallPackageSchema.index({ callSite: 1, createdAt: -1 });
providerCallPackageSchema.index({ outcome: 1, createdAt: -1 });
```

Do not add retention indexes or dashboard-oriented indexes in v0.1.

```js
  response: { ... },
  timing: { ... },
  outcome: 'success',
  error: null,
  redaction: { ... },
  storage: { ... }
}
```

Allowed v0.1 values:

- `direct-http`
- `gateway-http`
- `local-http`

- `cli`

### `operation`

Allowed v0.1 values:

- `image-parse`
- `chat`
- `parse-escalation`
- `transcribe-image`

Do not use this field to judge whether the model answer was correct. It is only call context.

### `outcome`

Allowed v0.1 values:

- `success`
- `http_error`
- `network_error`
- `timeout`
- `aborted`
- `invalid_json`

Do not use `recorder_error` as a provider call outcome. Recorder failure is about the harness, not the provider package. Recorder failure should be returned from the recorder helper and logged, but it should not replace the provider outcome.

## Request Package

Persist only redacted request headers. Never persist an unredacted copy of request headers.

```js
request: {
  method: 'POST',
  url: 'https://api.moonshot.ai/v1/chat/completions',
  protocol: 'https:',
  hostname: 'api.moonshot.ai',
  port: 443,
  path: '/v1/chat/completions',

  headers: {
    authorization: '[REDACTED]',
    'content-type': 'application/json',
    accept: 'application/json'
  },
  redactedHeaderNames: ['authorization'],

  bodyKind: 'json',
  bodyJson: { ... },
  bodyText: '{...}',
  bodyByteLength: 12345,
  bodySha256: '...',
  bodyPayloadRef: null,

  modelRequested: 'kimi-k2.5',
  timeoutMs: 30000
}
```

Rules:

- Store the exact JSON object before serialization when available.
- Store the serialized JSON text when safe to keep inline.
- Store byte length and SHA-256 for request body text.
- Store prompt text and image content by default when capture is enabled, because full package preservation is the goal.
- Treat prompt text and image content as sensitive stored evidence, not as auth secrets.
- Redact known secret headers before persistence.
- Redact secret-like body paths if they exist, such as `apiKey`, `accessToken`, `refreshToken`, `secret`, or `credential`.
- Never persist full process environment.
- Never persist secret file contents.

## Response Package

Persist the first server-visible HTTP response package.

```js
response: {
  received: true,

  statusCode: 200,
  statusMessage: 'OK',
  httpVersion: '1.1',

  headers: { ... },
  rawHeaders: ['content-type', 'application/json'],
  trailers: {},
  rawTrailers: [],

  bodyChunks: [
    {
      seq: 0,
      receivedAt: Date,
      byteLength: 1234,
      sha256: '...',
      text: '{...}',
      base64: null,
      payloadRef: null
    }
  ],

  rawBodyText: '{...}',
  rawBodyByteLength: 1234,
  rawBodySha256: '...',
  rawBodyPayloadRef: null,

  parsedJson: { ... },
  parseError: null
}
```

Rules:

- `bodyChunks[]` preserves ordered chunks as they arrived at Node's `IncomingMessage` boundary.
- `rawBodyText` preserves the complete response body string existing code effectively consumes today.
- `parsedJson` is an added preservation view. It must not replace `rawBodyText`.
- JSON parse failure should set `parseError` and must not change helper return behavior.
- Response headers are not expected to contain provider auth secrets, but run defensive redaction anyway.
- Redact `set-cookie`, `cookie`, and any response header with secret-like names.

## Timing Package

```js
timing: {
  requestStartedAt: Date,
  requestWrittenAt: Date | null,
  responseHeadersAt: Date | null,
  responseEndedAt: Date | null,
  recorderStartedAt: Date | null,
  recorderFinishedAt: Date | null,
  durationMs: Number | null,
  responseDurationMs: Number | null,
  recorderDurationMs: Number | null
}
```

Minimum required:

- request start
- request written
- response headers received when there is a response
- response end, error, timeout, or abort time
- total duration

## Error Package

```js
error: {
  kind: 'http_error',
  message: 'Provider returned HTTP 400',
  code: null,
  name: 'Error',
  stack: null,
  httpStatusCode: 400,
  rawBodyText: '{...}',
  parsedJson: { ... }
}
```

Rules:

- HTTP non-2xx responses should preserve the response package before provider code throws.
- Network errors should preserve request metadata and the Node error fields.
- Timeout and abort should be distinguishable.
- Invalid JSON should preserve raw body text and parse error.
- Recorder errors should be logged by the recorder and returned as `{ ok: false }`, not stored as the provider outcome.

## Redaction Rules

Add:

```text
server/src/services/provider-call-package-redaction.js
```

Minimum exported functions:

```js
redactHeaders(headers)
redactRawHeaders(rawHeaders)
redactProviderCallPackage(envelope)
```

Headers to redact case-insensitively:

- `authorization`
- `proxy-authorization`
- `x-api-key`
- `x-goog-api-key`
- `cookie`
- `set-cookie`
- any header whose name contains `api-key`
- any header whose name contains `token`
- any header whose name contains `secret`
- any header whose name contains `credential`

Allowed auth metadata:

- auth scheme, such as `Bearer`
- env var name, such as `MOONSHOT_API_KEY`
- boolean credential presence, such as `{ hasApiKey: true }`

Never store bearer tokens, API keys, session cookies, local auth tokens, full inherited process env, or secret file contents.

Redaction metadata:

```js
redaction: {
  applied: true,
  redactedHeaderNames: ['authorization'],
  redactedBodyPaths: [],
  notes: []
}
```

## Storage Rules

No silent truncation.

MongoDB has a 16 MB BSON document limit. Image requests can include large base64 payloads, so v0.1 must make storage behavior explicit.

Add:

```text
server/src/services/provider-call-package-payload-store.js
```

Use this runtime sidecar root:

```text
server/data/provider-call-packages/YYYY-MM-DD/<packageId>/<field-name>.txt
```

Reason: `server/data/` is already ignored by Git in this repo. The root-level `data/` path proposed by plan 2 is not currently ignored and should not be used unless `.gitignore` is updated.

Small inline payload:

```js
storage: {
  inline: true,
  externalPayloads: [],
  truncated: false,
  truncationReason: null
}
```

Externalized payload:

```js
storage: {
  inline: false,
  externalPayloads: [
    {
      field: 'request.bodyText',
      kind: 'request_body',
      byteLength: 123456,
      sha256: '...',
      encoding: 'utf8',
      ref: 'server/data/provider-call-packages/2026-05-20/<packageId>/request-body.txt'
    }
  ],
  truncated: false,
  truncationReason: null
}
```

If truncation ever happens:

```js
storage: {
  inline: false,
  externalPayloads: [],
  truncated: true,
  truncationReason: 'payload exceeded storage limit and no external storage was available'
}
```

v0.1 should avoid truncation. Prefer external reference or non-fatal recording failure over hidden truncation.

Implementation notes:

- Generate a package id before saving, using `new mongoose.Types.ObjectId()` or `crypto.randomUUID()`.
- Use that id in external payload paths.
- Include SHA-256 and byte length for every externalized payload.
- Do not create cleanup jobs or retention policy in v0.1.

## Feature Flag

Add to `server/.env.example`:

```text
ENABLE_PROVIDER_CALL_PACKAGE_CAPTURE=false
```

Behavior:

- Default false.
- When false, helpers behave exactly as they do today and no package records are written.
- When true, in-scope HTTP model calls attempt to write `ProviderCallPackage` records.
- Tests can enable it explicitly.
- The flag does not replace non-fatal recorder behavior.
- Even when enabled, recorder failure must not break provider calls.

Implementation helper:

```js
function isProviderCallPackageCaptureEnabled() {
  return String(process.env.ENABLE_PROVIDER_CALL_PACKAGE_CAPTURE || '').toLowerCase() === 'true';
}
```

## Recorder Service

Add:

```text
server/src/services/provider-call-package-recorder.js
```

Responsibilities:

- accept a complete envelope or HTTP capture input
- apply redaction before persistence
- externalize oversized payloads before Mongo write
- save `ProviderCallPackage`
- return `{ ok: true, id }` on success
- return `{ ok: false, error }` on failure
- log concise recorder failures
- never throw into provider call flow in normal mode

Recommended exports:

```js
recordProviderCallPackage(envelope, options)
buildHttpProviderCallPackage(input)
isProviderCallPackageCaptureEnabled()
```

Recorder policy:

- The HTTP helper may await the recorder inside a `try/catch` so tests can deterministically assert the record.
- Recorder failure must be swallowed after logging and returned internally.
- Recorder failure must not alter provider response text.
- Recorder failure must not alter provider usage.
- Recorder failure must not trigger fallback.
- Recorder failure must not change the HTTP status returned to the client.

## HTTP Capture Helper Design

Add optional capture metadata as the final argument to existing helpers.

For `image-parser.js`:

```js
jsonRequest(method, baseUrl, urlPath, body, headers, timeoutMs, captureContext)
```

For `remote-api-providers.js`:

```js
jsonRequestCancelable(method, baseUrl, urlPath, body, headers, timeoutMs, captureContext)
```

For `lm-studio.js`:

```js
rawRequest(baseUrl, method, urlPath, body, timeoutMs, extraHeaders, captureContext)
jsonRequest(method, urlPath, body, timeoutMs, extraHeaders, captureContext)
```

Capture context:

```js
{
  providerId: 'kimi',
  providerResearchId: 'kimi-api',
  providerPathType: 'direct-http',
  callSite: 'image-parser:callKimi',
  operation: 'image-parse',
  source: {
    file: 'server/src/services/image-parser.js',
    functionName: 'callKimi',
    helperName: 'jsonRequest'
  },
  modelRequested: effectiveModel
}
```

Rules:

- `captureContext` is optional.
- If omitted, helper behavior is unchanged.
- If the feature flag is off, helper behavior is unchanged.
- The helper must resolve or reject exactly as it does today.
- Existing call sites should not need parser changes.
- Capture should happen at the HTTP helper boundary, not inside parser extraction.

Capture flow:

1. Capture `requestStartedAt` before writing the request.
2. Build request metadata from the URL, method, headers, body, timeout, and capture context.
3. Serialize the payload exactly as today.
4. Capture `requestWrittenAt` after `req.write()` or before `req.end()` for empty bodies.
5. In the `transport.request(..., (res) => ...)` callback, capture:
   - `res.statusCode`
   - `res.statusMessage`
   - `res.httpVersion`
   - `res.headers`
   - `res.rawHeaders`
   - `responseHeadersAt`
6. On each `data` chunk, capture:
   - sequence number
   - received timestamp
   - byte length
   - SHA-256
   - UTF-8 text or external reference
7. Continue accumulating `data` exactly as today.
8. On `end`:
   - capture trailers and raw trailers when available
   - compute raw body SHA-256 and byte length
   - parse JSON best-effort
   - set `outcome` to `success`, `http_error`, or `invalid_json`
   - record non-fatally if capture is enabled
   - resolve the same object as today
9. On request error:
   - set `outcome` to `network_error` unless already classified as `timeout` or `aborted`
   - attempt non-fatal capture
   - reject the same error as today
10. On timeout:
   - set `outcome` to `timeout`
   - attempt non-fatal capture
   - reject the same timeout error shape as today
11. On cancel/abort in `jsonRequestCancelable(...)`:
   - set `outcome` to `aborted` when the helper can distinguish cancellation from ordinary network failure
   - preserve current cancellation behavior

## Risks And Mitigations

### Risk: Capture changes provider behavior

Mitigation:

- helper return shapes stay identical
- tests assert unchanged return values
- recorder failures are swallowed and logged non-fatally

### Risk: Secret leakage

Mitigation:

- centralized redaction helper
- persist only redacted headers
- redact request and response raw headers
- unit tests for common secret headers
- never store full process env
- never store secret file contents

### Risk: Mongo document too large

Mitigation:

- explicit payload store helper
- use `server/data/provider-call-packages/...` sidecar storage
- store byte length and SHA-256
- no silent truncation

### Risk: Too many provider paths change at once

Mitigation:

- Kimi proof first
- expand only after proof tests pass
- keep CLI, SDK, streaming, probes, and warm-ups out of v0.1

### Risk: Schema turns into analytics model

Mitigation:

- preserve raw package first
- only minimal indexes
- no parser-answer fields
- no UI/dashboard fields
- no retention policy in v0.1

### Risk: Worktree contamination

Mitigation:

- rerun `git status -sb`
- inspect dirty in-scope files before editing
- path-limit staging
- do not use `git add .`
- do not overwrite user changes

## Review Appendix: What I Changed Or Do Not Agree With

### Do Not Persist Unredacted Headers

Plan 2 includes `headersSent` plus `headersRedacted`. I do not agree with persisting an unredacted request-header copy. The final plan persists only redacted headers and redaction metadata.

### Use `server/data`, Not Root `data`

Plan 2 suggests `data/provider-call-packages/...`. Current `.gitignore` ignores `server/data/`, not root `data/`. The final plan uses `server/data/provider-call-packages/...` to avoid accidental untracked evidence files.

### Do Not Use `recorder_error` As Provider Outcome

Plan 2 includes `recorder_error`; plan 1 includes `recording_error`. I do not agree with making recorder failure a provider call outcome. The provider outcome should describe the provider attempt. Recorder failure should be logged and returned by the recorder as `{ ok: false }`.

### Prefer `provider-call-package-*` File Names

Plan 1 suggests `provider-package-recorder.js`. I prefer plan 2's naming style because it aligns with the model name and keeps the feature searchable:

- `provider-call-package-recorder.js`
- `provider-call-package-redaction.js`
- `provider-call-package-payload-store.js`

### Feature Flag Is Required For v0.1

Plan 1 treats the feature flag as optional. I do not agree. This capture stores prompts, images, raw provider bodies, and headers, so v0.1 should be explicitly enabled with `ENABLE_PROVIDER_CALL_PACKAGE_CAPTURE=true`.

### Update Plan 3, Not Plan 2

Plan 2 says to update `provider-harness-research/plan-2/v.01.md` after implementation. Since this document is the consolidated final plan, future status updates should go in plan 3 or a new implementation status doc, not back into plan 2.

### Use `parse-escalation`, Not `text-parse`

Plan 1 lists `text-parse`. The current source function is `parseEscalation`, so the final operation name is `parse-escalation`.

Both plans list all v0.1 paths, which is correct for final scope. The final implementation sequence is stricter: model/helpers first, Kimi proof second, then expansion only after proof tests pass.
