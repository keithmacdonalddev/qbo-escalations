# LM Studio Provider Harness End-to-End Plan v0.1

## Executive Decision

This plan changes direction from broad provider rollout to one provider first:

```text
Provider: LM Studio
App provider id: lm-studio
Research id: lm-studio-openai-compatible
Goal: make this one provider harness deterministic end to end before bringing other providers up to the same standard.
```

This is not a parser-quality plan. This is not a prompt plan. This is not a UI plan.

The only goal is:

```text
LM Studio provider attempt starts
-> this app sends the LM Studio request package
-> LM Studio sends a response/error/stream package
-> this app receives that exact package at the provider boundary
-> Mongo preserves that package in a provider-specific shape
-> existing app behavior continues unchanged
```

## Plain-English Definition

A provider harness is the plumbing around one provider call.

For LM Studio, the provider harness must preserve:

- what this app sent to LM Studio
- what LM Studio sent back
- how the response arrived
- whether the attempt succeeded, failed, timed out, ended without a `[DONE]` stream marker, or returned malformed JSON/SSE
- the timing and request/response boundary facts

The provider harness must not decide:

- whether the model answer is correct
- whether the parser output is good enough
- whether the prompt should be changed
- whether another provider should be tried
- whether canonical text should be built
- whether fields are missing or hallucinated

## Non-Negotiable Rules

1. LM Studio is the only provider in scope.
2. Provider package preservation is the only product goal.
3. Every in-scope LM Studio model-response attempt must create one LM Studio package record.
4. The record must preserve the provider package, not just the extracted model answer.
5. Provider calls must not wait for Mongo/file recording before returning the provider result to existing app code.
6. Recorder writes are fire-and-forget in production paths.
7. Tests may use a test-only recorder-settled hook.
8. Recorder failure must not fail the provider call.
9. Recorder failure must produce enough server-side log context to diagnose capture failure.
10. No parser validation changes.
11. No parser answer cleanup changes.
12. No prompt changes.
13. No fallback changes.
14. No UI changes.
15. No other provider harness changes.
16. No model-discovery capture in the first implementation pass.
17. No warm-up capture in the first implementation pass.
18. Do not add provider-harness development console breadcrumbs. Verification should come from captured records, focused tests, and explicit read-only inspection commands.
19. Any operational logging added later must never print full prompt text, base64 images, raw provider bodies, or secrets.
20. Mongo/file storage may externalize large payloads by reference, but it must not silently drop or truncate provider-sent response bodies.

## What "100%" Means Here

For this plan, "Mongo gets everything LM Studio sent" means:

- the full HTTP status and status message when an HTTP response exists
- response headers and raw headers
- response trailers and raw trailers when present
- ordered raw body chunks
- full response body text for non-streaming calls
- parsed JSON view when parseable
- JSON parse error when not parseable
- ordered streaming network chunks for streaming calls
- ordered SSE frames for streaming calls
- parsed SSE chunk JSON objects when parseable
- malformed SSE/JSON frame facts when not parseable
- `[DONE]` sentinel when present
- end-without-`[DONE]` fact when the stream ends without the sentinel
- full non-200 error body, not only the current 500-character error message
- network error facts when no HTTP response arrives
- timeout facts

The raw provider response body must not be transformed into canonical text at this layer.

Allowed safety exception:

- outgoing secret headers such as `Authorization` must be redacted before durable storage.

Payload-size rule:

- if a provider package is too large for safe Mongo inline storage, store a digest and external payload reference.
- the record must explicitly show where the payload was stored.
- no silent truncation.

## Required Storage Shape

Use the existing `ProviderCallPackage` collection as the outer envelope, but add a strict LM Studio-specific package section.

Decision:

```text
ProviderCallPackage.lmStudio is the source of truth for LM Studio package preservation.
```

Do not rely on loose `request: Mixed` / `response: Mixed` as the exact LM Studio database model.

Minimum top-level envelope:

```js
{
  schemaVersion,
  captureVersion,
  providerId: 'lm-studio',
  providerResearchId: 'lm-studio-openai-compatible',
  providerPathType: 'lm-studio-http-nonstream' | 'lm-studio-http-stream',
  callSite,
  operation,
  source,
  request: null | compatibilityView,
  response: null | compatibilityView,
  cli: null,
  lmStudio: { ...strictLmStudioPackage },
  timing,
  outcome,
  error,
  redaction,
  storage
}
```

Minimum `lmStudio` package:

```js
{
  mode: 'non-stream' | 'stream',

  request: {
    method,
    baseUrl,
    url,
    protocol,
    hostname,
    port,
    path,
    urlPath,
    headers,
    bodyKind,
    bodyText,
    bodyJson,
    bodyByteLength,
    bodySha256,
    modelRequested,
    stream,
    timeoutMs
  },

  response: {
    received,
    statusCode,
    statusMessage,
    httpVersion,
    headers,
    rawHeaders,
    trailers,
    rawTrailers,
    bodyChunks,
    bodyText,
    bodyByteLength,
    bodySha256,
    parsedJson,
    jsonParseError
  },

  stream: {
    rawChunks,
    frames,
    parsedChunks,
    doneSeen,
    terminator,
    finalBuffer,
    fullResponse,
    fullResponseByteLength,
    fullResponseSha256,
    usage
  },

  error: {
    name,
    message,
    code,
    stack,
    statusCode,
    rawBody,
    object
  }
}
```

Notes:

- `stream` may be `null` for non-streaming calls.
- `parsedJson`, `parsedChunks`, and error `object` may use `Mixed` internally because LM Studio follows OpenAI-compatible shapes but can add/omit fields.
- The strictness is around the LM Studio package container and known transport fields.
- The provider response itself must be preserved even when the model sends unexpected fields.

## Outcome Classification

Use these outcomes for LM Studio:

```text
success
http_error
network_error
timeout
aborted
invalid_json
malformed_sse
stream_end_without_done
recorder_error
```

Rules:

- HTTP status >= 400: `http_error`.
- No HTTP response and request error: `network_error`, unless timeout/abort applies.
- Timeout event: `timeout`.
- Abort/cleanup before completion: `aborted`.
- Non-streaming status 2xx with unparseable JSON when JSON was expected: `invalid_json`.
- Streaming frame with malformed JSON: preserve the frame and mark the frame parse error. Overall outcome remains `success` if the app still reaches success and `[DONE]`; use `malformed_sse` only if malformed stream content prevents successful completion.
- Stream ends without `[DONE]`: preserve as `stream_end_without_done` unless current app behavior treats it as success. If preserving current behavior requires top-level `success`, store `terminator: 'end_without_done'` inside `lmStudio.stream`.

## Recorder API

Add LM Studio-specific recorder functions:

```js
buildLmStudioProviderCallPackage(input)
recordLmStudioProviderCallPackage(input, options)
recordLmStudioProviderCallPackageInBackground(input, options)
```

Responsibilities:

- build the strict `lmStudio` envelope
- classify the outcome
- redact app-owned secrets
- externalize large payloads without truncating
- write to Mongo
- log recorder failures without failing the provider call
- expose a test-only settled hook already used by the shared recorder

Production code must call the background version.

Tests may await the settled hook.

## Redaction Policy

Redact:

- outgoing `Authorization`
- outgoing `Proxy-Authorization`
- outgoing cookies if they ever appear
- obvious API key/token/password fields in request JSON

Do not use redaction as answer cleanup.

Do not redact provider output merely because it is inconvenient, verbose, malformed, or hallucinated.

If response-body redaction ever triggers because a provider sent secret-like text, the record must show:

- redaction applied
- redacted path
- redaction note

## Payload Storage Policy

Mongo has a document size limit. LM Studio image requests and long streams can be large.

Rules:

- preserve small payloads inline
- externalize large payloads by digest/reference
- store byte length and SHA-256
- store the payload reference in the exact field where the payload would have been
- never silently truncate
- never store both a huge body text and the identical huge parsed JSON if they are byte-for-byte equivalent

LM Studio-specific externalization candidates:

- `lmStudio.request.bodyText`
- `lmStudio.request.bodyJson`
- `lmStudio.response.bodyText`
- `lmStudio.response.parsedJson`
- `lmStudio.response.bodyChunks[].text`
- `lmStudio.stream.rawChunks[].text`
- `lmStudio.stream.frames[].rawLine`
- `lmStudio.stream.frames[].data`
- `lmStudio.stream.parsedChunks`
- `lmStudio.stream.finalBuffer`
- `lmStudio.stream.fullResponse`
- `lmStudio.error.rawBody`

## Console Trace Policy

Provider-harness development console breadcrumbs were removed after testing. Do not add `providerHarnessTrace(...)`, `PROVIDER_HARNESS_CONSOLE_TRACE`, colored trace dependencies, or raw `console.log(...)` breadcrumbs for normal LM Studio harness flow.

Verification should use deterministic evidence instead:

- captured `ProviderCallPackage` records
- sidecar payload files and hashes
- focused tests
- explicit read-only Mongo/file inspection commands during manual testing

Operational logging may be added only if it is part of normal application error handling, and it must never print full prompt text, raw response text, raw stream text, base64 images, Authorization headers, full request bodies, or full response bodies.
