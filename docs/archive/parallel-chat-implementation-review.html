Overview

What Changed

The parallel chat UI was completely redesigned from stacked vertical responses to a side-by-side split-column layout on desktop with a segmented tab bar on mobile. Users can now accept a winner, optionally discard the loser, and track model performance across conversations. Three analysis features (Spotlight Diff, Conviction Strip, Snap Decision) appear for image-parse turns.

Winner / Loser Mechanics

Accept

Click anywhere on a card or press 1/2 to accept. Winner gets provider-colored border glow (.is-accepted) and an "Accepted" badge. The backend marks the winner with attemptMeta.accepted = true.

Discard (Optional)

After accepting, the loser shows a "Discard" button. Clicking it collapses the loser to a 48px slim bar (.is-discarded) with dashed border and 0.6 opacity. The "Click to restore" hint remains.

Keep Both

By default, both responses remain visible after acceptance. The loser gets subtle dimming (.is-rejected, 0.7 opacity) but stays fully readable. No action needed to keep both.

Re-enable

If discarded, clicking the slim bar restores the response. The discardedProviders state in Chat.jsx maps turnId → provider and is UI-only (not persisted to backend).

Persistent Split Mode

How It Works

Once parallel mode produces its first response, splitModeActive is set to true in useChat.js. This forces all subsequent sendMessage calls to use mode: 'parallel' regardless of the UI selector. The split view persists for the entire conversation.

  • Activation: Set in onDone handler when parallel results arrive
  • Persistence: Re-detected on conversation reload — selectConversation checks for any mode === 'parallel' messages
  • Reset: Cleared on newConversation()
Feature Accordion (Image Parse Only)

Three collapsible feature rows below the split outputs. Exclusive — only one expanded at a time. Only appears when the preceding user message had images (template parsing).

1. Spotlight Diff

Sentence-level semantic diff. Splits both responses into sentences, finds unique/shared via set comparison. Renders with colored left-borders: teal for A-unique, purple for B-unique, dimmed for shared. Computed lazily via useMemo.

2. Conviction Strip

Paragraph-level similarity as a horizontal stacked bar. Green = agree, amber = partial overlap, red = contradiction. Uses keyword Jaccard similarity per paragraph pair. Shows percentage agreement summary.

3. Snap Decision

Side-by-side metrics grid: response time A vs B, word count A vs B. The faster/more detailed response gets a green "Winner" highlight. Built in SnapDecision.jsx.

Model Performance Tracking

What Gets Tracked

Every parallel accept creates a ModelPerformance document with:

  • winnerProvider / loserProvider — which model won
  • winnerLatencyMs / loserLatencyMs — response times from ParallelCandidateTurn
  • winnerWordCount / loserWordCount — output length
  • context — "image-parse" or "general-chat" (auto-detected from preceding user message)
  • decidedAt — timestamp for time-series analysis

Analytics Endpoint

GET /api/analytics/model-performance returns per-provider win/loss stats, win rates, average latency, context breakdown, and daily time series. Uses MongoDB $facet for efficient single-query aggregation. Surfaced in the Analytics page with stat cards and a detailed breakdown table.

ModelPerformance Recording (non-blocking)
After updating conversation messages, a non-blocking ModelPerformance.create() call records the decision. It pulls latency from ParallelCandidateTurn.candidates[], computes word counts, and auto-detects context (image-parse vs general-chat) by checking if the preceding user message has images. Wrapped in try/catch so performance tracking never breaks the accept flow.
