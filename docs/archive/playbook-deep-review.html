Overall Verdict: Solid Engine, Unfinished Cockpit

The playbook's inference wiring is complete and well-designed — content loads into memory, injects as system prompt, feeds search retrieval and copilot analysis. The knowledge base content is mature (22 files, 2,589 lines, 9 categories, 6 templates). But the authoring UI is barebones, the backend has zero test coverage and security gaps, and critical management features (search, markdown preview, template editing, system-prompt editing) are missing entirely. Users today can browse and edit categories — but that's it. To exceed expectations, the playbook editor needs to become a first-class knowledge management tool, not a textarea wrapper.

Security & Reliability Issues

CRIT
No authentication on playbook endpoints
All 7 /api/playbook/* routes accept unauthenticated requests. Anyone with network access can read, modify, or delete playbook content. Acceptable for local-only use, but docs incorrectly claim auth exists.
server/src/routes/playbook.js · server/src/app.js:35
HIGH
Weak path traversal protection
safeName() strips special characters but doesn't enforce a strict allowlist. While the .md extension mitigates most attacks, the principle is weak. Should use UUID or [a-z0-9-] only.
server/src/routes/playbook.js:16-18
HIGH
No file size limit on uploads
readMarkdownFiles() has no max file size check. A malformed or malicious file could cause OOM. The 50MB Express body limit exists, but the loader reads from disk with no bounds.
server/src/lib/playbook-loader.js:25-43
MED
File operations lack error handling
writeFileSync() and unlinkSync() in PUT/DELETE handlers have no try/catch. Disk errors crash with unstructured 500 response.
server/src/routes/playbook.js:59, 94
MED
Filesystem watcher never closed
fs.watch() handle is never closed on server shutdown. On Windows, this can prevent directory deletion and accumulate file handles over time.
server/src/lib/playbook-loader.js:304-325
MED
No watcher debouncing
File watcher triggers buildSystemPrompt() on every change event. Rapid edits (10 saves in 1 second) cause 10 full rebuilds. Should debounce by 500ms.
server/src/lib/playbook-loader.js:311-318
LOW
Silent error in category stat
Empty catch block on line 29 swallows all stat errors. Permission errors cause categories to silently disappear from listings with no indication to the user.
server/src/routes/playbook.js:29
LOW
No recursive depth limit
readMarkdownFiles() recursively reads with no depth limit. Circular symlinks could hang the server indefinitely.

Recommended Build Roadmap

i
Priority philosophy: The inference pipeline works. The management layer is what's broken. Focus all effort on making the editor a complete knowledge management tool that covers all 22 files, not just 10.

Phase 1 — Unblock Full Content Management (Critical)

Close the biggest gap: make all 22 playbook files reachable through the UI.

  • 1 Template CRUD API — Add GET/PUT routes for templates/*.md files (mirror category pattern)
  • 2 Reference docs API — Add routes for top-level .md files (system-prompt, triage, error-messages, decision-tree, qbo-urls, translations)
  • 3 Redesign PlaybookEditor UI — Replace 3-tab layout with proper section-based navigation: Categories, Templates, Reference Docs, Full Prompt
  • 4 Wire template + reference UI — Add template and reference doc browsing/editing to the redesigned editor
  • 5 Fix error handling — try/catch on all file ops, specific error messages with retry, better save feedback

Phase 2 — Editor Quality (High Value)

Make the editor a pleasure to use, not just functional.

  • 1 Playbook search — Expose searchPlaybookChunks() to a /api/playbook/search endpoint, wire to search bar in UI
  • 2 Markdown preview — Toggle between raw text and rendered markdown with syntax highlighting
  • 3 Playbook stats dashboard — Total token estimate, per-section sizes, last modified dates, search chunk count
  • 4 Keyboard shortcuts — Ctrl+S save, Ctrl+E toggle edit, keyboard nav
  • 5 Draft auto-save — localStorage backup of in-progress edits to prevent data loss
  • 6 Unsaved changes protection — Warn on sidebar nav, not just page close

Phase 3 — Hardening & Polish

Backend reliability, security, and testing.

  • 1 Backend hardening — File size limits, watcher debouncing, watcher cleanup on shutdown, recursive depth limit
  • 2 Security — Strict path sanitization ([a-z0-9-] allowlist), input validation, content length limits
  • 3 Automated tests — Route integration tests, loader unit tests, search accuracy tests
  • 4 Accessibility — ARIA labels, semantic HTML, focus management, reduced motion support
  • 5 Mobile layout — Responsive sidebar collapse, touch-friendly edit controls
