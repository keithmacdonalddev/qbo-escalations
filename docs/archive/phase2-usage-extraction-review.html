1 — Executive Summary

What Phase 2 Does

Phase 2 modifies the two AI provider service files — claude.js and codex.js — to capture token usage data from CLI subprocess output. Usage data extracted via usage-extractor.js (Phase 1) is now threaded through every completion path: success, error, timeout, and abort. Downstream consumers (Phase 3 orchestrators, Phase 4 route handlers) can read usage from onDone's 2nd parameter, err._usage, cleanup() return value, or the { fields, usage } parse result wrapper.

2 — Usage Data Flow

Usage flows from CLI stdout through extraction, capture, and propagation. The capturedUsage variable uses last-write-wins — since the result event (which carries final usage) arrives last, this naturally captures the definitive value.

3 — claude.js Changes

chat() — 6 integration points

  • Import: extractClaudeUsage from ../lib/usage-extractor
  • State: let capturedUsage = null alongside fullResponse, killed, settled
  • Stdout handler: each parsed JSON line → extractClaudeUsage(msg) → overwrites capturedUsage
  • Close handler buffer: same extraction on final stdoutBuffer parse
  • finishWithSuccess: onDone(text, capturedUsage || null) Backward compat
  • finishWithError: err._usage = capturedUsage || null before onError(err)
  • cleanup(): returns { usage: capturedUsage, partialResponse: fullResponse }

parseEscalation() — Breaking return shape Breaking

  • State: let capturedUsage = null inside Promise scope
  • Close handler: extractClaudeUsage(parsed) on the full JSON output
  • ALL resolve(data) calls → resolve({ fields: data, usage: capturedUsage })
  • CLI error reject: cliErr._usage = capturedUsage || null
  • Timeout error: timeoutErr._usage = capturedUsage || null
  • Process error: err._usage = capturedUsage || null

6 — Key Design Decisions
Decision Rationale Impact
Last-write-wins for capturedUsage Each JSON line overwrites capturedUsage. The result event arrives last and carries definitive usage — so last-write naturally captures the correct value. No array accumulation needed. Simplicity
Backward-compat onDone 2nd param JS ignores extra function arguments. Callers that expect onDone(text) continue working — the 2nd param is silently dropped if unused. No regressions
err._usage convention Attaching usage to the Error object allows error handlers to capture partial token spend from failed/timed-out requests. The underscore signals it's a non-standard extension. Issue 1, 11
cleanup() return value Previously returned undefined. Now returns { usage, partialResponse } so route handlers can capture in-flight usage when a client disconnects mid-stream. Issue 11
{ fields, usage } breaking change parseEscalation() now returns a wrapper object. Phase 3's parse-orchestrator.js handles this with: const raw = rawResult.fields || rawResult — backward compat via duck-typing. Breaking
Double JSON parsing in codex.js Codex stdout handler parses each line for usage extraction AND then extractDeltaFromEventLine parses again internally. The overhead is negligible — codex events are small JSON objects. Avoids refactoring the delta extractor's API. Pragmatic
