Parallel Chat Redesign

A faster, more readable comparison UI for multi-provider AI responses — optimized for QBO escalation specialists on live phone calls.

Mika UX Research
Sora Visual Design
Kai Interaction Design

The Problem

The current parallel chat UI stacks responses vertically with poor visual separation. Specialists on live calls can't quickly scan, compare, or accept responses.

Poor Readability

Responses blend together visually. No color-coded provider identity. The bordered boxes all look the same, forcing the specialist to read provider labels to know which response is which.

Hard to Compare

Vertical stacking forces scrolling between responses. No visual connection between competing answers. The specialist must hold Response A in memory while scrolling to read Response B.

Tiny Action Buttons

Accept, Fork, and Copy buttons are small and positioned inconsistently. Under time pressure, Fitts's Law penalizes small targets — the specialist wastes precious seconds aiming.

No Difference Visibility

Both responses may agree on 80% of content. The specialist has no way to quickly see where they differ — they must read both fully, even when differences are minor.

Layout Decision: Hybrid Split-to-Tab

We evaluated 7 layout approaches. The unanimous recommendation is a Hybrid Split-to-Tab — side-by-side columns on desktop, segmented tabs on mobile.

Approach Scannability Readability Mobile Verdict
Side-by-Side Columns 9/10 7/10 3/10 Desktop pick
Tabbed / Segmented 4/10 10/10 10/10 Mobile pick
Card Carousel 3/10 9/10 9/10 Primacy bias
Stacked + Peek 5/10 7/10 7/10 Front-card bias
Master-Detail 6/10 8/10 5/10 Inherently biased
Accordion 5/10 6/10 8/10 Extra clicks
Overlay Slider 2/10 2/10 4/10 Incompatible with text

Why Hybrid Wins

  • Desktop (>900px): Side-by-side columns with synchronized scroll — instant visual comparison, no interaction required. Proven by LMSYS Chatbot Arena at 1M+ evaluations.
  • Mobile (<900px): Segmented control tabs — full-width readability with easy provider switching. Native to iOS/Android patterns.
  • Context Line bridge: A slim header between the user message and dual responses anchors the comparison in the chat flow: "2 responses — compare and accept one".
  • Provider color identity: Teal (Provider A) and purple (Provider B) top-borders + header tints make each response instantly identifiable without reading labels.

Interaction Model

Keyboard-first design with full-card click as mouse fallback. Optimized for sub-3-second decision time on live calls.

Primary: Keyboard Accept

Press 1 or 2 to accept left or right response. Number keys map spatially to position. Only active when parallel responses are displayed.

Speed: ~0.3 seconds

Secondary: Full-Card Click

Click anywhere on the card to accept it. Hover shows a provider-colored border glow as preview. Massive hit target eliminates Fitts's Law penalty.

Speed: ~0.8 seconds

Safety Net: 5s Undo

Gmail-style undo toast appears for 5 seconds after acceptance. Press Ctrl+Z or click "Undo" to reverse. Action isn't committed until the window expires.

Recovery: ~0.5 seconds

Post-Accept: Collapse + Integrate

Winner expands into a standard chat bubble. Loser collapses to a 36px bar: "[Provider] response — click to view". No information is lost.

Resolved in: 500ms total

Animation Sequence

500ms total from click to resolved state. Every millisecond is accounted for.

0ms — Trigger
User presses 1/2 or clicks a card
0–100ms — Flash Confirm
Winner card gets provider-color border glow (2px). Loser fades to 40% opacity. Instant visual confirmation that the action registered.
100–350ms — Resolve Layout
Winner card smoothly expands (ease-out). Loser collapses to a 36px bar: "[Provider] response — click to view". The conviction strip fades out.
350–500ms — Integrate
Winner card's glow fades, transitions to a standard chat bubble. Undo toast slides up from bottom. The response is now part of the conversation thread.
5000ms — Commit
Undo window expires. Toast fades out. The acceptance is persisted to the database. The collapsed loser bar remains accessible.

Feature 1: Spotlight Diff

Spotlight Diff

Proposed by Mika — UX Researcher

Semantic difference highlighting that saves 60–80% of reading time. Instead of reading both responses fully, the specialist sees at a glance where the responses diverge.

How differences are color-coded:

UNIQUE TO PROVIDER A

Content that only appears in Provider A's response (e.g., "same-day payroll options"). Teal highlight draws attention to A's unique recommendations.

UNIQUE TO PROVIDER B

Content that only appears in Provider B's response (e.g., specific QBO navigation path "Payroll > Employees > Payment Method"). Purple highlight for B's unique value.

FACTUAL DISAGREEMENT

Content where providers contradict each other (e.g., different processing timelines). Amber flags where the specialist must apply judgment — the most critical information.

AGREEMENT (NO HIGHLIGHT)

~80% of content where both providers agree. Unhighlighted = safe to skip. The specialist focuses only on the colored sections.

Toggle: Press D key or click "Show Diff" in the context line header. Highlights fade in over 300ms.

Implementation: Sentence-level semantic similarity computed server-side, or by prompting Claude to identify key differences as structured data.

Recommendation Summary

The Design Team's Unified Proposal

  • Layout: Hybrid Split-to-Tab — side-by-side columns on desktop (>900px), segmented tabs on mobile. Provider-colored headers (teal A, purple B) for instant identification.
  • Context Line: A slim bridge between user message and dual responses: "2 responses — compare and accept one" with provider color dots.
  • Interaction: Keyboard-first (1/2 to accept) with full-card click as mouse fallback. 5-second Gmail-style undo with Ctrl+Z.
  • Post-Accept: Winner expands into chat thread, loser collapses to a 36px "Show other" bar. 500ms total animation.
  • Feature 1 — Spotlight Diff (Mika): Semantic highlighting of unique content (teal/purple) and contradictions (amber). Saves 60-80% reading time.
  • Feature 2 — Conviction Strip (Sora): Vertical agreement/disagreement gradient gutter between panels. Sub-second orientation for where responses diverge.
  • Feature 3 — Snap Decision (Kai): Speed mode (S key) that compresses responses to one-line summaries for under-2-second acceptance.
