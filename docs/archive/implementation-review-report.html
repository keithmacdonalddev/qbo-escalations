Review Discussion Summary

Backend Eng: The server architecture is solid — clean route separation, proper model design, and the orchestrator pattern works well. But we have shell injection in every CLI spawn call, ReDoS in three places, and the copilot routes completely bypass provider independence. Those are show-stoppers.

Frontend Eng: Agreed on severity. On the client side, the "Warm Authority" design system is well-crafted, but there's a critical performance bug where motionProps invalidates useCallback on every render, messages use array index as keys (dangerous during retry), and both streaming hooks leak memory on unmount. The hooks share ~300 lines of duplicated code.

Backend Eng: The biggest architectural gap we both see: Phase 1 shipped without feature flags, which means there's no rollback path. Plus, parallel mode was implemented even though it's Phase 4 scope — without the Phase 4 test framework. That's risky scope creep.

Frontend Eng: And neither side has provider health visibility in the UI. The health module exists on the server but isn't wired into routing decisions or surfaced to the user. For a provider independence phase, that's a significant omission.

Backend Eng: Test coverage is also a concern. 50+ tests exist and cover the orchestrators well, but analytics (7 endpoints), copilot (8 endpoints), and playbook (7 endpoints) have zero tests. That's 22 untested endpoints.

Critical Issues

9 findings
CRIT-01 Command injection via shell: true in all CLI spawn calls Backend

server/src/services/claude.js:73-76   server/src/services/codex.js:70-74   server/src/routes/dev.js:288-293

All spawn() calls use shell: true. User-supplied message content is passed as positional CLI arguments. A prompt containing shell metacharacters (backticks, $(), &&, pipe) could be interpreted by the shell, enabling command injection.

const child = spawn('claude', args, {
  stdio: ['pipe', 'pipe', 'pipe'],
  shell: true,   // DANGEROUS — user input in args
  env: { ...process.env },
});
Recommended Fix: Remove shell: true from all spawn calls. Provide the full path to the CLI binary if needed. For codex.js which uses stdin, shell: true is entirely unnecessary.
CRIT-02 ReDoS vulnerability in template rendering — user keys injected into RegExp Backend

server/src/routes/templates.js:92-93

User-controlled key values from req.body.variables are interpolated directly into new RegExp() without escaping. An attacker can craft a key like (a+)+$ to trigger catastrophic backtracking, freezing the Node.js event loop.

rendered = rendered.replace(new RegExp('\\{\\{' + key + '\\}\\}', 'gi'), value);
rendered = rendered.replace(new RegExp('\\[' + key + '\\]', 'gi'), value);
Recommended Fix: Escape regex metacharacters from key before constructing the RegExp, or use String.prototype.replaceAll() with literal string matching.
CRIT-03 ReDoS in copilot search and escalation filtering via $regex Backend

server/src/routes/copilot.js:331-339   server/src/routes/escalations.js:176

User-supplied search query is passed directly into MongoDB's $regex operator without sanitization. Malicious regex patterns can hang the MongoDB query engine.

{ attemptingTo: { $regex: query, $options: 'i' } }
// and:
if (req.query.agent) filter.agentName = { $regex: req.query.agent, $options: 'i' };
Recommended Fix: Escape user input before using in $regex, or use MongoDB's $text search operator exclusively for text search operations.
CRIT-04 Copilot routes hardcoded to Claude — completely bypasses provider independence Backend

server/src/routes/copilot.js:5,28-58

All 8 copilot endpoints import claude directly and call claude.chat(). There is no provider parameter, no orchestrator usage, and no fallback. If Claude is down, all copilot features are completely unavailable — contradicting provider independence goals.

const claude = require('../services/claude');
// All endpoints:
cleanupFn = claude.chat({ ... });
Recommended Fix: Route copilot through the chat orchestrator or a dedicated copilot orchestrator. Accept provider/mode params on copilot endpoints.
CRIT-05 Dead parseEscalation in chatApi.js with wrong signature — will crash if imported Frontend

client/src/api/chatApi.js:148-164

The parseEscalation function expects a raw string and calls input.startsWith('data:image'). But callers pass an object with { text, image, mode } properties. The function is dead code (callers import from escalationsApi.js instead), but if anyone imports the wrong version, it will throw TypeError: input.startsWith is not a function.

Recommended Fix: Remove the dead parseEscalation export from chatApi.js to prevent accidental misuse.
CRIT-06 motionProps recreated every render — defeats useCallback memoization in App.jsx Frontend

client/src/App.jsx:49-56, 109

motionProps is an object literal created on every render. It's in the dependency array of useCallback for renderView, so the callback is invalidated every render cycle. This defeats the entire memoization strategy and causes AnimatePresence to re-mount all view components on every state change, causing layout jank.

Recommended Fix: Move motionProps to a useMemo with empty deps, or define it outside the component as a module-level constant (since it doesn't depend on props/state).
CRIT-07 No abort/cleanup on unmount in streaming hooks — memory leak during navigation Frontend

client/src/hooks/useChat.js:172-309   client/src/hooks/useDevChat.js:172-297

When the Chat or DevMode component unmounts while an SSE stream is active, neither hook aborts the in-flight request. Callbacks continue updating unmounted component state, causing React warnings and memory leaks. The abortRef.current is set but never called on cleanup return from useEffect.

Recommended Fix: Add a cleanup function to the useEffect that manages streaming, or add an unmountedRef guard. Call abortRef.current() in the hook's cleanup.
CRIT-08 Array index as React key for messages — causes reconciliation bugs on retry Frontend

client/src/components/Chat.jsx:497-527

Messages use key={i} (array index). When retryLastResponse pops messages and inserts new ones, React incorrectly matches new messages to old keys. This causes stale content, wrong copy button targets, and incorrect fork indices.

{messages.map((msg, i) => (
  <ChatMessage key={i} ... />
))}
Recommended Fix: Use key={msg._id || msg.timestamp || crypto.randomUUID()} for stable message identity.
CRIT-09 ~300 lines duplicated between useChat.js and useDevChat.js Frontend

client/src/hooks/useChat.js (540 lines)   client/src/hooks/useDevChat.js (350 lines)

60-70% identical logic: provider normalization, fallback resolution, localStorage persistence, streaming state management, abort handling, and conversation CRUD. Any bug fix to streaming or provider logic must be applied in both files, creating a maintenance hazard.

Recommended Fix: Extract shared logic into a useStreamingChat base hook. Specialized hooks only add their unique behavior (parallel for chat, tool events for dev).
B

Bugs

17 findings
BUG-01 Race condition: conversation save in parallel chat onDone — Mongoose full-doc replace Backend

server/src/routes/chat.js:282-334

The onDone callback pushes multiple assistant messages and calls conversation.save(). The conversation was fetched earlier (line 140). If another request modifies the same conversation between fetch and save, messages are lost due to Mongoose's full-document replacement.

Recommended Fix: Use atomic $push operations: Conversation.findByIdAndUpdate(id, { $push: { messages: newMsg } })
BUG-02 Graceful shutdown doesn't close HTTP server — SSE streams terminated abruptly Backend

server/src/index.js:46-52

The shutdown handler only closes MongoDB but never calls server.close(). Active SSE streams will lose data. In-flight requests get abruptly terminated.

Recommended Fix: Store HTTP server ref from app.listen(), call server.close() first, then close MongoDB.
BUG-03 Playbook categories listing crashes if file deleted between cache and statSync Backend

server/src/routes/playbook.js:22-28

getCategories() returns cached names. If a file was deleted since last cache build, fs.statSync(filePath) throws ENOENT and crashes the handler (Express returns 500).

Recommended Fix: Wrap statSync in try/catch or verify file existence before stat.
BUG-04 Codex system prompt treated as user content — worse playbook adherence Backend

server/src/services/codex.js:323-346

When systemPrompt is provided, buildPrompt prepends it as "System instructions:" in the prompt text. This means it counts against user token budget and may be treated as user-generated content by the model rather than as a system instruction — leading to worse QBO playbook adherence with Codex.

BUG-05 deriveSourceFromPayload crashes on null payload properties Backend

server/src/routes/escalations.js:33-44

Line 40 accesses payload.screenshotPaths without a payload && guard. If payload is null (broken content-type), this throws. Lines 34 and 37 have the guard, but line 40 does not.

BUG-06 DevConversation model missing 'parallel' in CHAT_MODES enum Backend

server/src/models/DevConversation.js:4

CHAT_MODES = ['single', 'fallback'] but resolvePolicy() can return mode: 'parallel'. If parallel is returned for a dev mode request, Mongoose will reject the save with a validation error.

BUG-07 Dev mode activeSessions map entries can leak on long-lived connections Backend

server/src/routes/dev.js:542-579

The cleanup function is called on req.on('close') and on success, but there's a window between cleanup and endStream where state is inconsistent. If the client keeps the connection alive indefinitely, the map entry persists.

BUG-08 Template picker modal onKeyDown will never fire — missing tabIndex Frontend

client/src/components/Chat.jsx:888-889

The modal overlay div has onKeyDown for Escape, but no tabIndex. Since div elements aren't focusable by default, Escape only works if a child element happens to bubble the event.

BUG-09 No debounce on dashboard search — fires API request on every keystroke Frontend

client/src/components/EscalationDashboard.jsx:104-110

Each keystroke fires listEscalations and getSummary API requests. No debouncing, creating excessive API calls and potential race conditions where older results overwrite newer ones.

BUG-10 Sidebar 10-second polling continues when tab is hidden Frontend

client/src/components/Sidebar.jsx:32-34

The sidebar polls conversations every 10 seconds via setInterval regardless of tab visibility. This wastes resources and creates unnecessary server load when the user is on another tab.

BUG-11 Potential XSS via markdown link javascript: URLs in ChatMessage Frontend

client/src/components/ChatMessage.jsx:267-268

The inlineFormat function renders <a href={match[8]}> from parsed markdown. If an AI response contains [click](javascript:alert(1)), it creates a javascript: URL. React warns but does not block it.

Recommended Fix: Validate that href starts with http:// or https:// before rendering as a link.
BUG-12 Inconsistent use of window.confirm() vs ConfirmModal Frontend

client/src/components/PlaybookEditor.jsx:155   client/src/components/TemplateLibrary.jsx:136

Both use window.confirm() for destructive deletes while the rest of the app uses the styled ConfirmModal component. Inconsistent UX and the native dialog blocks the main thread.

BUG-13 Stale closure in CopilotPanel handleRun — double-click fires two requests Frontend

client/src/components/CopilotPanel.jsx:42-44

handleRun is a plain function (not useCallback) that captures streaming state. Rapid double-clicks can fire two requests before streaming updates, because React batches state updates.

BUG-14 Sidebar reads window.location.hash directly — can be stale after navigation Frontend

client/src/App.jsx:139

currentRoute={window.location.hash || '#/chat'} reads the hash during render instead of using the route state. After a programmatic navigation, the currentRoute could be stale until the next hashchange event.

BUG-15 Unhandled promise rejections in onDone/onError SSE callbacks Backend

server/src/routes/chat.js:282-384

The onDone callback is async and does await conversation.save(). If save throws (validation error, connection lost), the rejection is uncaught. The SSE stream hangs without sending done/error events.

Recommended Fix: Wrap onDone body in try/catch that sends an error SSE event on failure.
BUG-16 conversationId stale closure in removeConversation Frontend

client/src/hooks/useChat.js:455-463

removeConversation depends on conversationId state value captured at callback creation time. If it changes between creation and invocation, it compares against a stale value. Should use conversationIdRef.current.

BUG-17 Memory leak: setTimeout callbacks in copy buttons never cleaned up on unmount Frontend

client/src/components/ChatMessage.jsx:315   DevMode.jsx:523   PlaybookEditor.jsx:342

setTimeout(() => setCopied(false), 2000) is never cleaned up. If the component unmounts within 2 seconds, setState targets unmounted state.

Security Concerns

5 findings
SEC-01 No authentication on any endpoint — gap matrix claim of authz is inaccurate Backend

The gap matrix states "Role-based API key auth exists for protected routes (server/src/middleware/authz.js)" but no authz.js middleware file was found. All endpoints (CRUD, delete, playbook editing, dev mode file access) are completely open. Acceptable for local-only use per CLAUDE.md, but the documentation is misleading.

SEC-02 Dev mode file-read endpoint exposes .env files Backend

server/src/routes/dev.js:815-849

/api/dev/file reads any file within PROJECT_ROOT. While isPathWithinRoot prevents directory traversal above the project, .env, server/.env, and other secrets within the project tree are fully accessible.

Recommended Fix: Add exclusion list for .env*, *.key, *.pem patterns in the file-read handler.
SEC-03 CORS completely open — any origin can call the API Backend

server/src/app.js:16

app.use(cors()) allows any origin. Acceptable for localhost but a vulnerability if the server is ever exposed to a network.

SEC-04 Static uploads served without access control Backend

server/src/app.js:17

All uploaded escalation screenshots are publicly accessible at /uploads/escalations/<id>/<hash>.jpg. No access control or authorization check.

SEC-05 No global Express error handler — unhandled errors return HTML instead of JSON Backend

server/src/app.js

Express 5 auto-catches async errors, but with no fallback error handler, unhandled errors return Express's default HTML 500 response instead of the { ok: false, code, error } JSON format the app uses.

Recommended Fix: Add a final app.use((err, req, res, next) => { ... }) middleware returning consistent JSON errors.

Omissions vs Phase 1 Plan

11 findings
OM-01 Missing chat-provider.js formal adapter interface with validators Backend

Phase 1 plan (line 55-58) specifies server/src/services/providers/chat-provider.js with a formal contract and validators. This file does not exist. The registry references claude.chat and codex.chat directly without contract validation.

OM-02 Missing feature flags — no rollback mechanism for provider independence Both

Phase 1 specifies FEATURE_CHAT_PROVIDER_PARITY and FEATURE_CHAT_FALLBACK_MODE flags. Neither exists in the codebase. Features are always-on with no way to disable provider independence features if they cause issues. This removes the planned rollback safety net.

OM-03 Missing structured server-side observability logging Backend

Phase 1 plan (lines 179-191) calls for structured per-turn logging with mode, providerUsed, fallbackUsed, latencyMs, etc. Metadata is persisted in MongoDB but there is no structured logging to stdout/stderr. Only console.log for startup events.

OM-04 Provider health not integrated into routing decisions Backend

Phase 1 plan: "Prefer healthy provider in fallback mode." The provider-health.js module tracks health and has a circuit breaker, but the chat and parse orchestrators never consult health state. An unhealthy provider is still tried first, wasting time on a known-bad path.

OM-05 No image size cap on chat endpoint Backend

Phase 1 plan: "Cap images per request and image bytes." The escalation route has MAX_RAW_IMAGE_BYTES = 20MB, but the chat route passes images through to the CLI provider with no size validation.

OM-06 Missing dedicated chat fallback integration test file Backend

Phase 1 plan specifies server/test/chat-fallback-integration.test.js. Integration tests exist in integration-routes.test.js but there's no dedicated fallback-focused test file with the scenarios listed in the plan.

OM-07 No provider health UI surface Frontend

Phase 1 specifies "Provider health visibility baseline." The frontend has no health indicator, no circuit-breaker status display, and no way for users to see if a provider is degraded before selecting it. The backend health endpoint exists but isn't consumed.

OM-08 No turn-level observability UI for provenance metadata Frontend

Phase 1 specifies per-turn observability: mode, requestedPrimaryProvider, providerUsed, fallbackUsed, fallbackReasonCode, latencyMs, errorCode. The UI shows provider label and response time but not attempt count, reason codes, or attemptMeta.

OM-09 Retry button doesn't support different provider/mode than current selection Frontend

Phase 1: "Retry endpoint with fallback policy." The retry button passes the current provider but doesn't expose UI to retry with a different provider or mode. Uses implicit refs, which works but limits user control.

OM-10 Parallel mode implemented despite being Phase 4 scope — shipped without Phase 4 tests Both

Phase 1 explicitly lists "Parallel opinions" as out of scope. However, parallel mode is fully implemented in both UI and API. This scope creep shipped without the Phase 4 acceptance framework, test suite, or cost controls that were planned to accompany it.

OM-11 Gap matrix claims auth middleware exists — it doesn't Backend

Gap matrix states: "Role-based API key auth exists for protected routes (server/src/middleware/authz.js)." No authz.js file was found on disk. The documentation is inaccurate and gives a false sense of security.

Architecture Issues

5 findings
ARCH-01 ~250 lines duplicated between chat POST and chat retry handlers Backend

server/src/routes/chat.js:108-392 vs 638-886

The retry handler is a near-complete copy of the main chat handler — identical SSE setup, orchestration callbacks, parallel turn creation, onDone/onError logic. Any bug fix must be applied in both places.

Recommended Fix: Extract shared orchestration-to-SSE logic into a handleChatStream() helper function used by both handlers.
ARCH-02 Duplicated utility functions across 5+ files Backend

parsePositiveInt in claude.js, codex.js, dev.js • isPathWithinRoot in escalations.js, dev.js • didCliExitSuccessfully in claude.js, codex.js, dev.js • normalizeProviderError in chat-orchestrator.js, parse-orchestrator.js, dev.js

Recommended Fix: Create server/src/lib/utils.js for shared helpers. Import everywhere.
ARCH-03 Provider registry constructs fresh objects on every getProvider() call Backend

server/src/services/providers/registry.js:47-58

Every getProvider() invocation calls def.getChat(), def.getDefaultTimeoutMs() etc. and creates a new object. The orchestrators call this multiple times per request. Not a crash bug but unnecessary overhead.

ARCH-04 No shared SSE helper between chat, copilot, and dev routes Backend

Each route file independently implements SSE headers, heartbeats, and event writing. Copilot has its own initSSE() and streamClaude() helpers, but these aren't reused. Inconsistencies in heartbeat intervals and cleanup behavior across routes.

ARCH-05 ~100 inline style objects across components instead of CSS classes Frontend

Chat.jsx (~30 inline styles), EscalationDetail.jsx (~20), Analytics.jsx (~15), and others define styles inline. This prevents hover states, media query overrides, and causes unnecessary re-renders (new object references each render). Inconsistent with the well-structured CSS design system in App.css.

UI/UX & Accessibility Issues

15 findings
UX-01No loading indicator when selecting a conversation from sidebar

Clicking a conversation triggers a fetch with no loading state shown. User sees nothing happen until fetch completes, which could take seconds.

UX-02Empty state quick prompts ("Parse Escalation") confuse AI with no context

Chat.jsx:453-462 — Quick prompts like "Parse Escalation" send parsing instructions with no content, confusing the AI. Should be disabled or replaced with contextual prompts when no conversation exists.

UX-03No pagination — capped at 50 items with no "load more"

EscalationDashboard and Sidebar fetch max 50 items. No pagination controls, no infinite scroll, no indication that more items exist.

UX-04Two-column layouts not responsive on mobile/tablet

EscalationDetail.jsx:180 — gridTemplateColumns: '1fr 1fr' hardcoded, won't collapse to single column on narrow screens. Content becomes unusable on mobile.

UX-05No unsaved-changes warning when navigating away from PlaybookEditor

PlaybookEditor tracks hasUnsavedChanges but never sets beforeunload or intercepts navigation. Users can lose edits by clicking a sidebar link.

UX-06Template picker modal lacks focus trap and keyboard management

Chat.jsx:885-980 — Tab key moves focus behind the overlay. No initial focus management. Not a proper role="dialog" with aria-modal="true".

UX-07No visual distinction between Quick Parse and AI Parse results

Both parse paths show provider info in the parseMeta banner, but there's no clear quality indication to help users decide whether to re-parse with AI vs. accept the quick regex parse.

A11Y-01No skip-to-content link for keyboard/screen reader navigation

The app has a sidebar with multiple links but no "skip to main content" link, forcing keyboard users to tab through all navigation items on every page.

A11Y-02Chat messages have no ARIA live region — new messages not announced

New messages and streaming text updates are invisible to screen readers. The chat-messages div should have aria-live="polite".

A11Y-03Filter selects in EscalationDashboard missing labels

EscalationDashboard.jsx:86-110 — Status and category <select> elements have no aria-label or associated <label>. Screen readers can't identify their purpose.

A11Y-04Color-only differentiation for category badges

Category badges use background color alone to differentiate categories. Users with color vision deficiency cannot distinguish between them. Should add icons or text patterns.

A11Y-05Sidebar conversation items have button-within-button semantics

Sidebar.jsx:169-275 — Each conversation item is role="button" with edit/delete buttons nested inside. This creates invalid ARIA semantics that confuse screen readers.

A11Y-06No keyboard shortcut hints via aria-keyshortcuts

Ctrl+N and Enter shortcuts are described in visible text but not as aria-keyshortcuts attributes.

A11Y-07Template picker overlay not an accessible dialog

Missing role="dialog" and aria-modal="true". Compare with ConfirmModal which correctly uses these attributes.

Performance Concerns

11 findings
PERF-01Full conversation messages array loaded on every chat requestBackend

server/src/routes/chat.js:140 — Every chat request loads the entire conversation document (all messages) with Conversation.findById(). For long conversations, this is hundreds of KB loaded, serialized to CLI, then re-saved.

PERF-02No cap on conversation message array growth — 16MB MongoDB doc limitBackend

Messages are an unbounded array in the conversation document. Hundreds of long messages could approach MongoDB's 16MB document size limit and cause write failures.

PERF-03buildHashesFromPaths reads all screenshot files synchronouslyBackend

server/src/routes/escalations.js:118-132 — Reads every existing screenshot file with fs.readFileSync(). For escalations with many screenshots, this blocks the event loop.

PERF-04Playbook loaded synchronously on first request — blocks event loopBackend

server/src/lib/playbook-loader.js:118-124 — First call to getSystemPrompt() synchronously reads all playbook files from disk. Blocks event loop during first request.

PERF-05Escalation text search query has no pagination cap on full-text scanBackend

server/src/routes/escalations.js:177-179 — $text search combined with sort and skip can cause full index scans on large datasets.

PERF-06Entire message list re-renders on every streaming chunkFrontend

Every setStreamingText call re-renders the Chat component and all ChatMessage children (index keys prevent React from skipping any). For long conversations with streaming, this causes significant jank.

Recommended Fix: Wrap ChatMessage in React.memo. Use stable message keys.
PERF-07Markdown re-parsed on every streaming chunk (50+ times/second)Frontend

ChatMessage.jsx:33-36 — useMemo([role, content]) invalidates on every chunk during streaming, triggering full markdown reparse. For large responses, this is heavy computation every frame.

PERF-08Analytics fires 8 parallel API calls on every mount — no cachingFrontend

Analytics.jsx:46-70 — 8 parallel fetch calls on every mount. Navigate away and back = 8 more calls. No stale-while-revalidate or caching strategy.

PERF-09Large base64 images stored permanently in React stateFrontend

Base64-encoded images (several MB each) are kept in the messages array in React state. Over a long session with many image-attached messages, memory usage grows significantly with no cleanup.

PERF-10No code splitting / lazy loading — all views eagerly importedFrontend

All views imported eagerly in App.jsx. No React.lazy + Suspense. The initial bundle includes Analytics, PlaybookEditor, DevMode, etc. even if the user only needs Chat.

Recommended Fix: Add lazy loading for rarely-used views. Can cut initial bundle by 30-40%.
PERF-11No request deduplication or caching across the API layerFrontend

Sidebar polls every 10s, dashboard reloads on every filter change, analytics fires 8 calls on mount. Switching views and returning reloads everything from scratch. http.js is a bare passthrough with no caching layer.

Special Features That Could Be Added

12 recommendations
FEAT-01Keyboard command palette (Ctrl+K) for quick actionsFrontend

A command palette for: new conversation, search escalations, switch views, change provider, insert template. Would significantly speed up the escalation specialist workflow.

FEAT-02Provider comparison dashboard for parallel mode quality analysisBoth

With parallel mode storing both provider responses, build a comparison view to evaluate which provider produces better QBO escalation guidance over time. Track acceptance rates per provider.

FEAT-03Conversation summarization before trimming for long chatsBackend

When conversations get long, use the AI to generate a summary of earlier messages, then trim originals. Preserves context while keeping token usage manageable.

The CSS defines .toast-container and .toast classes, but no toast component exists. Would replace scattered inline error/success messages with a consistent notification pattern.

FEAT-05Auto-save drafts to localStorageFrontend

Save the current input textarea content to localStorage so it survives page refreshes and accidental navigations. Critical for escalation specialists writing long messages.

FEAT-06Escalation timeline view for workflow bottleneck identificationFrontend

Replace/supplement the table in EscalationDashboard with a timeline showing status transitions over time. Helps identify bottlenecks in the escalation workflow.

FEAT-07Offline/connection status indicatorFrontend

Show a banner when the connection to the server is lost. Critical for escalation specialists who need to know if their work might not be saved.

FEAT-08Webhook/notification on escalation status changesBackend

Notify when an escalation moves to "resolved" or "escalated-further". Valuable for team visibility even in a primarily single-user tool.

FEAT-09Automated playbook gap detection from resolution dataBackend

The copilot has a playbook-check endpoint. Automate it to run periodically and flag categories with low resolution rates, surfacing playbook content gaps.

FEAT-10Provider cost tracking for parallel mode spend visibilityBackend

The gap matrix mentions cost should be trackable. Adding estimated token counts per provider response enables cost visibility, especially important when parallel mode doubles API usage.

FEAT-11Bulk operations on escalation dashboard (multi-select + batch actions)Frontend

Add checkboxes for multi-select and bulk status changes (e.g., resolve all selected). Escalation specialists often process batches of similar cases.

FEAT-12Provider health indicator next to provider selectors in UIFrontend

Show a small health dot (green/yellow/red) next to provider selectors based on GET /api/health/providers. Helps users avoid selecting a degraded provider.
